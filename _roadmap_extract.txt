# Prioritized Task Management Roadmap — Clone vs ClickUp

Roadmap ordered by **user impact ÷ effort**, aligned with your current stack (`backend-py` + Next.js). Assumes chat (Phase PY-4/5) stays stable; task work builds on Phase 3.

**P0 (before any release):** Fix the build error in `ChatSocketProvider.tsx` — `workspaceId` is `string | null` but `applyWorkspaceMemberJoined` expects `string | undefined`. One-line fix: pass `workspaceId ?? undefined` or widen the helper type.

## Phase 4A — Task hygiene (foundation)
**Goal:** Make tasks feel complete for daily use.  
**Effort:** ~1 week total  
**ClickUp gap closed:** ~15%

| # | Feature | Size | Backend | Frontend | Acceptance criteria |
|---|---------|------|---------|----------|---------------------|
| 4A.1 | **Delete task** | S | `DELETE /tasks/{id}` in [`home.py`](backend-py/app/api/v1/home.py), `delete_task` in [`home_service.py`](backend-py/app/services/home_service.py) | Delete in `TaskDrawer`, confirm dialog | Task removed from list/board/calendar; gone from search |
| 4A.2 | **Move task between lists** | M | `PATCH /tasks/{id}` add `listId` or `POST /tasks/{id}/move` | List picker in drawer; optional drag in sidebar | Task appears in new list only |
| 4A.3 | **Priority in API + UI** | S | Add `priority` to `UpdateTaskBody` in [`schemas/home.py`](backend-py/app/schemas/home.py) | Priority select in [`TaskDrawer.tsx`](frontend/src/components/spaces/TaskDrawer.tsx); badge on `TaskRow` | Urgent/high/normal/low set and shown |
| 4A.4 | **Add comment in drawer** | S | Already exists: `POST /tasks/{id}/comments` | Comment input + list in `TaskDrawer` | User can post without link-task flow |
| 4A.5 | **Edit on task detail page** | S | — | Reuse drawer form on [`TaskDetailView.tsx`](frontend/src/components/home/TaskDetailView.tsx) or redirect to list+drawer | `/home/tasks/[id]` is not read-only |
| 4A.6 | **Backend tests** | M | `tests/test_tasks.py` — CRUD, move, delete, filters | — | CI covers task APIs |

**Key files:** `home_service.py`, `home.py`, `TaskDrawer.tsx`, `spaces.ts`, `home.ts`

## Phase 4C — Workflow: statuses & tags
**Goal:** Per-list workflows like ClickUp (biggest UX jump for PM users).  
**Effort:** ~2 weeks  
**ClickUp gap closed:** +20%

| # | Feature | Size | Design |
|---|---------|------|--------|
| 4C.1 | **Custom statuses per List** | L | New `ListStatus` table: `list_id`, `name`, `color`, `group` (open/active/done/closed), `sort_order`. Migrate off fixed `TaskStatus` enum OR map enum → custom labels per list |
| 4C.2 | **Status inheritance** | M | Optional: List inherits from Space default statuses (ClickUp pattern) |
| 4C.3 | **Closed / archived status** | S | `CLOSED` group; hide from board by default; “Show closed” toggle |
| 4C.4 | **Tags** | M | `TaskTag` + `Tag` tables; multi-select on task; filter in list view |
| 4C.5 | **Start date** | S | `start_date` on `Task`; show in drawer + calendar range |

**Migration note:** Board columns today come from [`task-status.ts`](frontend/src/lib/task-status.ts) — replace with API-driven columns from list meta.

```mermaid
flowchart LR
    ListMeta --> ListStatuses[ListStatus API]
    ListStatuses --> BoardView
    ListStatuses --> TaskDrawer
    Task --> status_id[statusId FK]
```

## What NOT to build early

| Feature | Why wait |
|---------|----------|
| Full Gantt | Needs dependencies + start dates + stable statuses first |
| Full automation catalog | High complexity; 5 rules cover 80% of value |
| Goals/OKRs | Separate product surface; tasks aren’t mature enough |
| Whiteboards / Docs | Chat + tasks still higher ROI |
| 15 views | List + Board + Calendar + Timeline is enough for M3 |
